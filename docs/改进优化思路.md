# DELI-Fusion 外部建议审计与落地决策

审计日期：2026-07-15

本文后半部分保留 ChatGPT 和豆包的原始评价。本节将这些建议与当前代码逐项对齐，给出
“接受、修正、暂缓”的工程决策。结论是：两份评价对路由消融、持续自适应、长期退化和
内存拆解的判断基本正确，但不能为了让方法更像 learned index 就直接加入一个简单的
`z -> block id` 模型。

## 1. 直接接受的建议

以下四项是当前最高优先级：

1. **路由消融与可观测性**：增加 `FORCE_SFC_PATH`、`FORCE_GLOBAL_GUARD_PATH` 和
   `ADAPTIVE_ROUTE`，并统计每条路径的 query 数、probe 数、切换次数、候选数、exact
   calls 和实际 CPU 时间。若 global guard 长期承担超过 95% 的查询，论文不能把性能归因
   于 SFC overlay 或 learned routing。
2. **滚动自适应路由**：当前 4--8 次 warm-up 后永久选路只是一次性选择。应周期性抽样
   备用路径，用 EMA、最小样本数、非对称 hysteresis 和 cooldown 支持 workload drift；
   路由探测开销必须单独计账。
3. **长期 stale/delta 控制**：为 Base SpatialGuard 增加代价驱动重建，并在 5M--10M
   mixed operations 下报告 stale ratio、delta size、query drift、rebuild CPU 和尾延迟。
   不能把 lazy deletion 的维护成本隐藏在实验区间之外。
4. **内存和更新消融**：拆分 geometry、metadata、SFC blocks、base guard、delta guard、
   bitmap 和 local guard 的内存；比较 Boost、Boost + lazy delete、Fusion without bitmap
   和 Full Fusion，明确查询、更新、内存之间的 Pareto trade-off。

## 2. 需要修正的建议

### 2.1 PRL 不是质心捷径

当前代码调用 `Geometry::getCoordinate()`。GEOS 的接口契约是“返回该 Geometry 的一个
vertex，空 geometry 返回空指针”，不是 centroid 或 envelope center。因此：

```text
geometry vertex inside query rectangle => geometry intersects query rectangle
```

该正向 shortcut 是 exact-safe 的，不应删除。需要做的是：

- 论文和结构图统一改称 **geometry-vertex witness**；
- 保留 `representative_point` 参数名仅用于脚本/CSV 兼容；
- 增加 point、line、concave polygon、polygon with holes、geometry collection、empty
  geometry 和 boundary-touch 的专项正确性测试。

### 2.2 简单 learned block router 不是当前 P0

“增加 `z -> block id` 模型，否则不像 learned index”是叙事驱动，而不是问题驱动。当前
对象是 extent `(zmin, zmax)`，查询条件是 interval overlap。一个只预测 `qz -> block`
的位置模型无法安全跳过更早开始但 `zmax` 很大的 extent；若没有 conservative correction
和层级 `max_zmax/MBR`，它可能漏报。即使只把它用于 `lower_bound` 加速，它优化的也主要是
目录定位，而不是当前占主导的候选生成和 GEOS exact cost。

此外，global guard fast path 开启后根本不访问 SFC directory。此时加入 learned router
不会改善主查询路径，只会增加结构、训练和论文解释成本。正确顺序是先证明 SFC path 在
某些 workload 下确实有独立优势，再决定是否加入 **extent-safe learned directory**。

如果后续实现，模型必须预测有序 block boundary，并配合有界修正、prefix/subtree
`max_zmax` 和 MBR conservative pruning；不能仅凭预测位置作为扫描起点。

## 3. 暂缓的建议

以下机制应在 profile 证明瓶颈后再做：

- 外接圆等 negative predicate：MBR disjoint 已是安全拒绝层，新增摘要只有在显著减少
  exact calls 且收益超过额外计算/内存时才值得保留；
- 强化 local SpatialGuard：当前 global path 已提供空间剪枝，应先用消融判断 local guard
  是否冗余；
- 全局 SFC repartition 和元数据压缩：有长期 drift 或 memory breakdown 证据后再实现；
- Z-cover/micro-segment：这是让 SFC path 真正具备竞争力的候选方向，但应单独消融，不能
  与路由、重建和 learned directory 一次性叠加。

## 4. 推荐实施顺序与验收门槛

### Phase 0：先回答“优势到底来自哪里”

实现三种强制路由模式、route-share/cost counters、PRL 顶点见证测试和 memory breakdown。

实现状态（2026-07-15）：已完成。代码现已支持 `FUSION_ROUTE_MODE=adaptive|force_sfc|force_guard`
和独立的 `FUSION_UPDATE_MODE=coupled|full|light`，并输出 route/probe/candidate/exact/time
分账及组件级内存估算。另新增独立结构消融 `DELI_FUSION_GUARD_ONLY`，彻底关闭 SFC
构建、block 定位和 per-block delta 维护，仅保留 Base/Delta R-tree、live bitmap 与 PRL。
复现步骤见 `docs/DELI-Fusion-Phase0复现.md`。

验收门槛：相同数据布局和 workload 下完成 Force SFC、Force Guard、Adaptive、Guard-only
与 Boost R-tree 五组消融；Guard-only 的所有 SFC 内存字段必须为 0；
正式抽样实验使用 `CHECK_CORRECTNESS=1`，所有 `answers_match_boost=1`。

### Phase 1：把一次性开关改为真正自适应

实现周期 re-probe、EMA、hysteresis、cooldown 和 workload-drift 实验。路由决策使用实测
端到端成本校准，避免只比较 block checks 或 R-tree visits。

验收门槛：在 read/write/selectivity phase change 中，Adaptive 能回切，并在计入 probe
开销后接近每个阶段较优的静态路径。

### Phase 2：约束长期退化

实现 cost-driven base guard rebuild。触发条件应比较预测的未来 stale/delta 查询损失与
实测 rebuild 成本，而不是只使用固定 20% 阈值。

验收门槛：5M--10M 操作中 query latency 不随 stale ratio 无界增长；重建成本、暂停时间、
峰值内存和摊销成本全部进入结果。

### Phase 3：增强 SFC path 后再决定 learned router

先分别测试 micro-segment、Z-cover 和 extent-aware block summaries。只有 Force SFC 在
高选择性、顺序扫描友好或写密集 workload 下能够胜过 Force Guard，双路径协同才有实证
基础。此后再实现带保守修正的 learned directory，并与 binary-search directory 消融。

## 5. 当前论文叙事

在上述证据完成前，最稳妥的定位是：

> DELI-Fusion 是面向动态非点对象的 predicate-cost-aware dual-route spatial index。它以
> cost-partitioned SFC overlay 和 update-light Base/Delta SpatialGuard 协同查询，并通过
> live bitmap、single-delta maintenance 与安全谓词精化，在查询延迟、更新吞吐和内存间
> 提供可测量的折中。

不要现阶段声称“learned extent router”已经实现，也不要把 global guard 的收益归因于
learned model。先完成 Phase 0 的归因实验，再决定论文最终强调 hybrid spatial index，还是
继续投入真正的 extent-safe learned routing。

## 最终判断

当前 DELI-Fusion **比之前版本更接近可发表系统**，甚至在工程性能上已经有亮点。但目前论文最大风险不是性能，而是：

> 它容易被审稿人重新定义为“Boost R-tree + adaptive wrapper”，而不是 learned spatial index。

最关键的一刀不是继续增加功能，而是：

**加入一个轻量、可解释、低成本的 learned routing component，并证明 Fusion 的优势来自“learned extent routing + cost-aware hybrid execution”，而不是 global R-tree fast path。**

这是我认为当前版本从“性能不错的系统”提升到“VLDB故事”的关键修改。




## 豆包原始评价

豆包评价意见：这份DELI-Fusion的设计整体框架自洽、性能目标明确，核心的「双路径架构+谓词成本对齐+轻量动态维护」思路成立，已经具备顶会论文的方法完整度。但从**学术严谨性、审稿人质疑预判、长期运行鲁棒性、贡献边界清晰度**四个维度看，仍存在不同优先级的短板，我按风险等级从高到低拆解，并给出对应的修改方案。

## 一、P0级：必须修正的核心风险（直接影响正确性与核心贡献可信度）

### 1. 自适应路由机制的「一次性决策」缺陷，与「自适应」定位不符

#### 问题现状

当前路由仅在warm-up阶段用4~8次探测就永久选定路径，开启global fast path后不再回测SFC路径，本质是**静态一次性开关**，而非真正的工作负载自适应。

- 风险1：负载漂移（如从小查询切到大查询、从读密集切到写密集）时，路由无法回切，SFC路径的优化完全失效；
- 风险2：warm-up样本量极小，容易受缓存预热、初始数据分布、偶然噪声影响，决策准确性无保障；
- 风险3：开启全局快路径后，块模式调整、局部重校准、SFC统计等机制全部被绕过，SFC路径沦为摆设，双路径架构的价值大打折扣。

#### 修改方案

升级为**带滞回的滚动窗口自适应路由**：

1. 保留5%/10%的非对称滞回阈值，避免频繁切换；
2. 每执行N次查询（如1000次），强制探测一次备用路径，更新两条路径的滑动平均耗时；
3. 全局快路径开启后，仍保留SFC路径的轻量统计（不中断主流程），保证维护策略持续有效；
4. 论文中明确区分「启动期暖身路由」和「运行期动态自适应」，补充不同负载漂移场景下的路由切换实验。

### 2. PRL代表点捷径的保守性定义模糊，存在正确性隐患

#### 问题现状

文档中第二条PRL捷径仅写为「对象的一个真实representative coordinate位于查询矩形内则必相交」，但未明确定义代表点的性质：

- 如果是质心/重心：凹多边形、带洞面、曲线的质心可能落在对象本体之外，会产生假阳性（判成交际不相交），违反保守性不变式；
- 没有对应的正确性证明，是审稿人必然会追问的正确性漏洞。

#### 修改方案

1. **明确定义+补充证明**：将代表点严格限定为「对象几何的边界顶点」，给出形式化证明：若多边形/线的任意一个顶点位于查询矩形内部，则对象与矩形必然存在公共点，接受判定零假阳性；
2. 移除模糊表述，在正确性不变式中新增PRL专项条款：所有接受型捷径均有严格的几何保守性证明；
3. 若保留质心方案，必须降级为「启发式预过滤」，不得纳入PRL的安全接受层。

### 3. 全局空间守卫的长期退化无约束，性能边界不可控

#### 问题现状

Base SpatialGuard是静态批量加载的，删除仅修改live位图、不修改树结构，插入仅进入Delta Guard，无代价驱动的重建机制：

- 长期运行下，base树的stale条目占比、delta树的规模会持续膨胀，最终查询性能可能反超原生R树，且退化程度不可预测；
- 文档仅提到「监控stale ratio」，没有对应的应对策略，动态索引的长期稳定性无法保障。

#### 修改方案

1. 增加**代价驱动的Base Guard重建机制**：当stale ratio + delta size超过阈值（如总对象数的20%），且估算重建收益大于重建成本时，在预算内异步/批量重建base树，合并delta、清理stale条目；
2. 补充**长时间负载退化实验**：跑500万~1000万次混合操作，对比Fusion与原生R树的性能漂移曲线，证明退化可控、重建机制有效；
3. 论文中主动说明懒删除的trade-off：删除吞吐大幅提升的代价是有限的stale开销，且该开销可通过预算式重建控制。

### 4. 缺失核心路由消融，无法证明双路径的协同价值

#### 问题现状

当前仅提供Full Fusion的整体性能，若不做路由拆分，审稿人必然会质疑：「你的性能优势是不是完全来自Boost R-tree的全局快路径？本质就是R-tree加了个懒删除位图？」

- 若强制走全局守卫的版本和Full Fusion性能几乎一致，SFC路径的所有设计都会被认为是冗余的。

#### 修改方案

必须补充三组核心路由消融，作为实验的核心支撑：

| 消融版本                     | 配置                                      | 验证目标                                                        |
| ---------------------------- | ----------------------------------------- | --------------------------------------------------------------- |
| Force SFC Path               | 强制走SFC块路径，关闭全局守卫             | 验证SFC架构本身的性能基线，以及代价划分、块模式等优化的独立价值 |
| Force Global Guard Path      | 强制走base+delta全局守卫路径，关闭SFC路由 | 验证纯轻量R树方案的性能上限，剥离SFC的贡献                      |
| Adaptive Route (Full Fusion) | 默认自适应路由                            | 验证自适应选择带来的额外收益                                    |

通过三组对比，可清晰证明：

- 小查询场景全局守卫更优，大查询/写密集场景SFC路径更优；
- 自适应路由能在不同负载下自动逼近最优性能，实现1+1>2的协同效果。

## 二、P1级：建议完善的严谨性短板（提升说服力，避免审稿争议）

### 1. 贡献边界模糊，易被弱化「换皮R树」

#### 问题现状

全局守卫直接复用Boost R-tree，若叙事重心放在「查询性能超越R树」，很容易被质疑是「用R树赢R树」，核心贡献被稀释。

- 当前设计的最大优势其实是**更新性能（尤其是删除）的碾压级提升**：文档中Fusion删除吞吐是Boost R-tree的近4倍，插入吞吐也高14%，但叙事重心偏查询，没有突出这个核心差异化。

#### 修改方案

1. **调整核心叙事重心**：从「查询更快的空间索引」转为「兼顾查询性能与极高更新吞吐的双路径自适应索引」，重点突出懒删除+delta守卫带来的更新性能优势，查询持平/略优只是附加收益；
2. **明确双路径的协同分工**：
   - 全局守卫路径：负责低选择性小查询的低延迟查询，更新成本极低（懒删除+追加delta）；
   - SFC块路径：负责高选择性大查询、写密集场景的高吞吐扫描与更新；
   - 自适应路由：自动匹配负载最优路径，实现全场景综合性能最优。
3. 主动对比「纯R树+懒删除」方案，证明SFC路径在大查询、写密集场景的不可替代性。

### 2. 代价模型的闭环性不足，「成本感知」说服力弱

#### 问题现状

- 构建时的DP划分仅用128条采样查询估算成本，段内记录用64个样本外推，非均匀数据下误差较大；
- 代价模型的系数（c_scan、c_mbr、c_exact等）未说明校准方式，若为人工预设常数，「predicate-cost-aware」的严谨性会打折扣；
- 运行时的维护、路由决策没有和全局代价模型打通，成本感知仅停留在构建阶段。

#### 修改方案

1. 补充**代价系数校准流程**：启动阶段通过微基准测试实测当前硬件下的单条扫描、单次MBR判断、单次GEOS调用的耗时，作为模型系数，而非固定常数；
2. 优化采样精度：非均匀数据集下增加采样密度，或用分层采样保证估算准确性；
3. 打通运行时代价反馈：将运行时统计的扫描量、候选数、GEOS耗时反馈给块维护策略，动态调整beta/tau阈值，实现全流程的代价驱动。

### 3. 内存开销无拆解，trade-off表述不清晰

#### 问题现状

Fusion内存比Boost R-tree高56.8%，比SingleStore-Cost高近3倍，但文档没有拆解内存构成，也没有主动说明性能-内存的取舍，容易被审稿人攻击「靠堆内存换性能」。

#### 修改方案

1. **内存拆解量化**：明确各组件的内存占比：SFC块目录、全局守卫（base+delta）、元数据、几何存储分别占多少；
2. 补充**性能-内存帕累托对比**：用散点图展示所有方法的查询吞吐vs内存、综合吞吐vs内存，主动说明Fusion的取舍：以有限的内存增量，换取数倍的更新吞吐提升与持平/更优的查询性能；
3. 增加轻量配置选项：如关闭局部守卫、降低delta阈值，提供内存优先的配置，丰富trade-off维度。

### 4. 局部空间守卫（Local Guard）定位冗余，收益不明确

#### 问题现状

块内可选的局部R-tree设计非常鸡肋：

- 仅在热块构建，且是前台同线程执行，不是后台无开销优化；
- 收益未验证，和micro-segment剪枝的功能重叠度高，反而增加了系统复杂度。

#### 修改方案

二选一定位：

1. **弱化路线**：将其移出核心架构，作为可选增强项放入消融实验，证明边际收益有限，不纳入主方法；
2. **强化路线**：和micro-segment整合，形成「块摘要→微段摘要→局部R树」的三级块内剪枝，纳入代价模型统一调度，明确其适用场景（如超大块、高碎片块）。

## 三、P2级：可选优化的提升点（完善方法完整性，拔高贡献）

### 1. 补全PRL双向保守谓词框架

当前PRL只有接受型捷径，没有拒绝型，框架完整性不足。可补充**对象外接圆-查询矩形不相交**的安全拒绝型谓词，形成「拒绝→接受→精确」的完整分层管线，同时纳入代价模型做最优排序，强化「谓词感知精化层」的学术贡献，而非单个优化技巧。

### 2. SFC路径补充Z-Cover与微段剪枝，缩小与全局守卫的查询差距

当前SFC路径的查询性能弱于全局守卫，导致自适应路由大多时候走R树路径，双路径的协同性不足。给SFC路径加上之前验证过的**有界Z多区间覆盖+块内micro-segment剪枝**，提升低选择率查询下的SFC性能，让两条路径各有胜负，自适应的价值才能真正体现。

### 3. 增加全局代价驱动的SFC重划分

当前仅支持单块局部重校准，没有全局块重新划分机制，长期数据分布漂移后，初始的DP划分会失效。可增加预算式的全局重划分机制，强化「全生命周期自适应」的特性。

### 4. 元数据紧凑化优化

所有object id、block id统一用uint32_t，MBR坐标用int32_t缩放存储，位图替代布尔数组，进一步压缩内存开销，缩小和原生R树的内存差距。

---

## 四、整体修改优先级与核心叙事建议

### 落地顺序

1. 先补P0级的4项：路由机制升级、PRL正确性补全、长期退化实验、核心路由消融——这是保证方法站得住脚的基础；
2. 再做P1级的叙事调整、代价模型完善、内存拆解——提升论文的严谨性和说服力；
3. 最后按需补充P2级的优化，进一步拔高方法完整度。

### 核心贡献的稳妥表述

调整后的贡献定位，完全避开「换皮R树」的争议，同时突出差异化：

1. 指出动态非点空间查询的核心成本是「扫描+候选精化+精确谓词+更新维护」的联合开销，而非单一一维查找成本；
2. 提出谓词成本感知的双路径索引架构，结合代价划分的SFC块覆盖与轻量base-delta空间守卫，兼顾查询效率与极高的更新吞吐；
3. 设计工作负载自适应的路由机制与块级多模式维护策略，在不改变查询语义的前提下实现读写性能的动态平衡；
4. 通过多数据集、多负载的实验验证，该方法在综合吞吐、删除性能上显著优于传统与学习型基线，同时保持具有竞争力的查询延迟与内存开销。
