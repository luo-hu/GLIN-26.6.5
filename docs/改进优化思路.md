这个结果完全在预期之内，**算不上“不妙”，只是PRL这个通用优化把“GEOS瓶颈”给打掉了，让竞争从「精化开销」回到了「索引原生结构开销」的赛道上**——这恰恰印证了你之前的判断：当结构剪枝足够有效后，瓶颈会转移；而当精化瓶颈被PRL解决后，原生结构的遍历、维护开销就成了新的主要矛盾。

R树作为发展了40年的经典结构，原生删除、遍历效率本来就极强；DELI目前的双结构设计、元数据开销、维护路径长度，在纯删除吞吐上暂时吃亏是正常的。我们不需要追求“所有指标全面碾压R树”（系统论文从来不是这个评判标准），而是要做三件事：

1. **叙事上摆正定位**：讲清楚trade-off，突出DELI的核心差异化价值，不硬拼单点指标；
2. **工程上精准优化**：按优先级砍掉冗余开销，把删除/查询的原生性能提上来，缩小差距；
3. **实验上立体呈现**：拆分维度，把DELI的优势（候选质量、长期稳定性、内存效率、GEOS削减率）充分展示出来。

## 一、先做核心判断：为什么R树+PRL反超了？

先把本质原因想透，优化才能打在七寸上：

1. **GEOS开销被大幅抹平了**：之前查询延迟里GEOS占80%+，DELI靠更少的候选量+PRL的高接受率拉开差距；现在PRL把所有方法的GEOS调用都砍掉了80%~90%，剩下的开销主要是「索引遍历+候选扫描+MBR比较」，这正是R树最擅长的领域。
2. **DELI当前有双结构冗余开销**：ALEX写入层 + compact查询块两套结构并存，插入删除要维护两份元数据，路径更长；而R树是单树结构，删除直接在节点内操作，原生路径更短。
3. **删除场景R树天然占优**：R树的删除是节点内原地标记+树内调整，经过几十年工程优化；DELI需要维护tombstone、block摘要、delta区、对象表反向映射，路径更长，写放大更高。

但这并不代表DELI没有价值——**它的核心优势从来不是“纯删除速度最快”，而是「动态更新下的性能有界性、代价可解释的自适应能力、结构-精化协同的GEOS削减率、学习型布局的内存紧凑性」**。我们只要把这些优势讲透，同时把原生性能差距缩小到可接受范围，就完全站得住脚。

## 二、论文叙事层面：主动出击，不硬拼单点

这部分最关键，直接决定审稿人会不会觉得“你这个工作不如R树”。核心原则是：**不回避R树的优势，主动定义清楚我们的贡献边界和差异化价值**。

### 1. 明确定位：我们不是“更快的R树替代品”

在引言和相关工作里主动写清楚：

> R树等传统空间索引在原生遍历与动态更新上经过了数十年工程优化，在简单几何、小查询场景下具有极强的端到端性能。本文的目标不是全面替代R树，而是提出一种新的设计范式：将学习型有序布局与保守谓词分层精化相结合，在复杂几何精确查询场景下，提供可证明的正确性、可解释的代价自适应、以及极具竞争力的综合性能。

### 2. 强化协同效应：DELI让PRL发挥了更大作用

你一定要把**「各方法加PRL后的GEOS削减率」**单独拎出来对比，这是DELI的核心差异化证据：

- DELI-Cost + PRL：exact calls 削减 95.6%
- Boost R-tree + PRL：exact calls 削减 X%（大概率在70%~85%之间）
- GLIN + PRL：exact calls 削减 Y%

结论非常硬：

> PRL作为通用精化层对所有方法均有效，但由于DELI结构层输出的候选集具有更高的空间聚集性与真阳率，PRL在DELI上的精确调用削减幅度显著更高。在更复杂的几何场景（如带洞多边形、长线路径）下，单次GEOS调用成本会指数级上升，这种协同优势将转化为更显著的端到端收益。

这句话直接把“大家都能用PRL”的质疑，变成了“DELI能让PRL效果更好”的贡献。

### 3. 把“删除吞吐”放进综合权衡框架里

不要单独比删除速度，要讲**读写综合性能 + 长期稳定性**：

- 增加「混合负载综合吞吐」指标（比如 query_tps × 0.7 + insert_tps × 0.15 + delete_tps × 0.15），展示综合权衡优势；
- 增加「长期性能漂移」曲线：跑满50万甚至200万次操作后，看R树会不会因为节点碎片、重叠增多而性能持续下降，而DELI靠局部压实始终维持稳定的性能边界。
  - 这刚好对应你论文里的「动态更新代价有界性定理」，用实验支撑理论，说服力极强。

## 三、工程优化层面：按优先级落地，精准砍开销

你列的优化方向全部正确，我帮你排好优先级，标注收益大小和实现难度，你可以按顺序落地：

### 优先级1：合并双结构冗余（收益最大，结构性优化）

**当前最大的开销来源**：ALEX写入层和compact查询块存了两份数据，插入、删除都要走两条路径，元数据重复维护，写放大高。

#### 优化方案：单主存储 + 轻量模型预测层

- 把**compact query blocks作为唯一主存储**，所有extent entry、delta、tombstone都只存在block里；
- 砍掉ALEX的完整leaf数组，只保留ALEX的**模型节点层**（即Z-key到block id的预测模型），作用只有一个：快速定位插入/查询对应的目标block。
- 这样：
  - 插入：模型预测block → 写入block的delta区 → 扩张block摘要，不用再写一份ALEX leaf；
  - 删除：对象表定位block → 标记tombstone，不用同步删除ALEX中的条目；
  - 查询：模型预测起始block → 顺序扫描block，和现在逻辑一致。

**收益**：删除和插入吞吐预计提升30%~50%，内存占用减少20%+，完全符合你“写入布局与查询块解耦”的原始设计，只是去掉了冗余的数据副本。

### 优先级2：简化删除路径，减少无效操作（收益高，实现简单）

当前删除路径大概率做了很多不必要的工作，按保守性原则，很多操作可以延后到compaction时做。

1. **删除默认不刷新block摘要**
   删除只会让block内的live集合变小，旧的stale-large摘要仍然满足保守不变式，完全安全。只有触发local compaction时，才精确重建block摘要。
   这一条就能砍掉删除路径里很大一部分计算开销。
2. **精简tombstone存储**
   不用在每个entry里加alive字段，而是在block header里维护一个**紧凑的tombstone id列表/bitmap**，扫描时通过bitmap判断是否存活。
   好处：entry体积更小，cache命中率更高；删除只需要改bitmap，不用触碰entry数据。
3. **弱化反向索引开销**
   object_to_block映射只存uint32_t的block id，不要存多余状态；删除时只定位block，不做多余的状态更新。

### 优先级3：元数据压缩 + 内存紧凑化（收益中，实现简单）

1. **全链路ID压缩**：所有object id、record id、block id，只要规模在400万以内，全部用 `uint32_t`，不用 `uint64_t`，直接砍掉一半元数据空间。
2. **MBR坐标压缩**：如果是投影坐标系（米制），可以用 `int32_t`存储（乘以1000缩放因子保留毫米精度），比double省75%空间；block summary体积缩小，cache友好度大幅提升。
3. **结构体紧凑排列**：调整extent entry的字段顺序，消除结构体padding；对block summary、entry结构体启用 `__attribute__((packed))`，减少内存浪费。

**收益**：内存占用下降，同时扫描时cache命中率提升，查询延迟可降低10%~20%。

### 优先级4：查询扫描的Cache友好性优化（收益中，针对性强）

现在GEOS开销降下来了，查询瓶颈转移到内存访问上，cache友好度直接决定速度。

1. **合并MBR判断与PRL判断**：读一次对象MBR，同时完成「是否相交」和「是否被包含」两个判断，避免两次内存读取。
2. **delta区连续存储**：把block的delta区也放在连续内存块里，不要用动态vector分散分配，保证扫描时是顺序内存访问，触发硬件预取。
3. **预取优化**：扫描主区和delta区时，提前若干条目做软件预取，掩盖内存延迟。

### 优先级5：精简Cost版的统计与压实开销（收益中，针对性强）

DELI-Cost比LB慢，主要是统计维护和更频繁的compaction导致的。

1. **统计采样更新**：block级的q_B/i_B/d_B访问频率，不用每次查询/更新都精确计数，可以用滑动窗口采样计数，减少写操作。
2. **压实触发滞后**：β、τ阈值不要卡得太死，允许超出10%~20%再触发compaction，用少量扫描开销换取更少的压实次数，提升整体吞吐。
3. **运行时禁用全局重划分**：DP最优划分只在bulk load时执行一次，运行时只做局部分裂/合并，保证运行时代价和LB版本基本持平。
